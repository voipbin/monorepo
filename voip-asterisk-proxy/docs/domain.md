# voip-asterisk-proxy — Domain

## Domain Entities

### Asterisk instance identity

Each proxy instance identifies itself using the MAC address of a configured network interface (`--interface_name`, default `eth0`). This MAC address serves as the Asterisk ID across the platform:

| Artifact | Value | Storage |
|----------|-------|---------|
| Asterisk ID | MAC address (e.g., `42:01:0a:a4:0f:d0`) | Kubernetes pod annotation `asterisk-id`; volatile RabbitMQ queue name |
| Internal address | IPv4 address of same interface | Redis key `asterisk.<id>.address-internal`, TTL 24 hours, refreshed every 5 minutes |

Upstream services use the Asterisk ID to route RPC requests to the correct proxy instance via the volatile queue.

### ARI event

An event message published to RabbitMQ whenever Asterisk emits an ARI WebSocket event. Format:

```json
{
  "type": "ari_event",
  "data_type": "application/json",
  "data": "<Asterisk ARI event JSON>"
}
```

The proxy forwards all ARI events without filtering. Consumers (call-manager, conference-manager, etc.) inspect the `type` field within `data` to decide relevance.

### AMI event

An event message published to RabbitMQ from the AMI TCP connection:

```json
{
  "type": "ami_event",
  "data_type": "application/json",
  "data": "<AMI event key=value pairs as JSON>"
}
```

Optional AMI event filtering is supported via `--ami_event_filter` (comma-separated event type names). An empty filter passes all events.

### RPC request / response

Incoming RabbitMQ RPC request envelope:

```json
{
  "uri": "/ari/channels?...",
  "method": "POST",
  "data": "<request body>",
  "data_type": "text/plain"
}
```

Response envelope returned to the caller's reply queue:

```json
{
  "status_code": 200,
  "data_type": "application/json",
  "data": "<Asterisk response body>"
}
```

Error responses use standard HTTP status codes (e.g., 400 for unrecognized URI, 500 for Asterisk communication failure).

### Recording file

A call recording generated by Asterisk, stored on the local filesystem at `--recording_asterisk_directory` (default `/var/spool/asterisk/recording`). On a `/proxy/recording_file_move` request, the proxy uploads the file to GCS bucket `--recording_bucket_name` under `--recording_bucket_directory`, then removes the local copy. Authentication uses Application Default Credentials (ADC).

## Key Business Rules

1. **Identity is MAC-based, not hostname-based.** The MAC address of the configured network interface is the stable identity. In Kubernetes, pod names change on restarts but MAC addresses are consistent within a pod's network namespace lifetime.

2. **All ARI events are forwarded.** The proxy does not filter ARI events by type — it publishes everything. This keeps proxy logic minimal; consumers are responsible for relevance decisions.

3. **Auto-reconnect on ARI/AMI failure.** Both the ARI WebSocket and AMI TCP connections implement 1-second retry loops. A transient Asterisk restart does not require proxy restart.

4. **RPC is synchronous.** The proxy blocks the RabbitMQ consumer goroutine until Asterisk responds to the ARI HTTP request or AMI action. Long Asterisk operations will hold the consumer for their duration.

5. **Kubernetes integration is optional.** Setting `--kubernetes_disabled=true` skips pod annotation patching. This is used in non-Kubernetes environments or during local development.

6. **Redis address registration is best-effort.** If Redis is unavailable, the proxy logs the error and continues. Address registration failure does not crash the service, but upstream services that use Redis to look up the proxy address will be unable to route to it.
