# Design: Configurable VAD Parameters

## Problem

Voice Activity Detection (VAD) in pipecat-manager is hardcoded with a single constant (`defaultVADStopSecs = 0.5`). Users cannot tune VAD behavior per-AI agent. Pipecat's `VADParams` supports 4 parameters (`confidence`, `start_secs`, `stop_secs`, `min_volume`), but only `stop_secs` is set, and it overrides Pipecat's default (0.2) with 0.5 without user control.

## Approach

Introduce a `VADConfig` struct in the AI model package, add it as a field on both the AI and AIcall models, and remove the hardcoded override so Pipecat's native defaults apply when no config is set.

## Data Model

### New struct: `ai.VADConfig`

Location: `bin-ai-manager/models/ai/main.go`

```go
type VADConfig struct {
    Confidence *float64 `json:"confidence,omitempty"`
    StartSecs  *float64 `json:"start_secs,omitempty"`
    StopSecs   *float64 `json:"stop_secs,omitempty"`
    MinVolume  *float64 `json:"min_volume,omitempty"`
}
```

All fields are optional pointer types (`*float64`). A nil pointer means "use Pipecat default". Using pointers instead of bare `float64` allows users to explicitly set `0.0` for any parameter ‚Äî bare `float64` with `omitempty` would silently drop `0.0` values during JSON serialization.

Pipecat defaults are:
- `confidence`: 0.7
- `start_secs`: 0.2
- `stop_secs`: 0.2
- `min_volume`: 0.6

### AI model

Add field after `STTType`:

```go
VADConfig *VADConfig `json:"vad_config,omitempty" db:"vad_config,json"`
```

### AIcall model

Add snapshot field after `AISTTType`:

```go
AIVADConfig *ai.VADConfig `json:"ai_vad_config,omitempty" db:"ai_vad_config,json"`
```

Snapshotted from `ai.VADConfig` at AIcall creation in `aicallhandler/db.go`, following the same pattern as `AIEngineModel`, `AITTSType`, etc.

## Database Migration

New Alembic migration:

```sql
ALTER TABLE ai_ais ADD vad_config JSON DEFAULT NULL AFTER stt_type;
ALTER TABLE ai_aicalls ADD ai_vad_config JSON DEFAULT NULL AFTER ai_stt_type;
```

## Python VADParams Construction

```python
def build_vad_params(vad_config: dict | None) -> VADParams:
    if not vad_config:
        return VADParams()  # Pipecat defaults

    kwargs = {}
    if vad_config.get("confidence") is not None:
        kwargs["confidence"] = vad_config["confidence"]
    if vad_config.get("start_secs") is not None:
        kwargs["start_secs"] = vad_config["start_secs"]
    if vad_config.get("stop_secs") is not None:
        kwargs["stop_secs"] = vad_config["stop_secs"]
    if vad_config.get("min_volume") is not None:
        kwargs["min_volume"] = vad_config["min_volume"]

    return VADParams(**kwargs)
```

Note: Uses `is not None` checks instead of truthiness to correctly handle explicit `0.0` values.

## Team Pipeline

For team pipelines, `vad_config` comes from the AIcall snapshot (which reflects the start member's AI or the team-level merged parameters). Each team member's `resolvedAIData` also carries `vad_config` from their individual AI, but the pipeline-level VAD uses the AIcall's snapshotted value since there is one VAD analyzer per input transport.

## What Gets Removed

- `defaultVADStopSecs = 0.5` constant in `run.go`
- `vad_stop_secs: float = 0.5` default in Python `PipelineRequest`
- `max(vad_stop_secs, 0.3)` safety floor in `run.py`
- All `vad_stop_secs` float parameters throughout the Go/Python call chain (replaced by `vad_config` struct/dict)

## Backward Compatibility

- Existing AI records will have `vad_config = NULL`, meaning Pipecat defaults apply (stop_secs=0.2 instead of the old hardcoded 0.5).
- This is intentional ‚Äî the user wants to remove the override and use Pipecat's native defaults when no config is set.
- Users who want the old 0.5s behavior can set `vad_config: {"stop_secs": 0.5}` on their AI.
