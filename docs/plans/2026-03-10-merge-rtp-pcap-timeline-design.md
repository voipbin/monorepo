# Merge RTP PCAP into Timeline PCAP Response

**Date:** 2026-03-10
**Status:** Draft

## Problem Statement

The timeline pcap endpoint (`GET /timelines/calls/{call_id}/pcap`) currently returns only SIP signaling (hepid=1) and RTCP stats (hepid=5) from Homer. RTP media pcap files exist in GCS (`rtp-recordings/<sip-call-id>-<ssrc>.pcap`) but are not included. Users debugging call quality need the full picture — signaling and media together in one pcap.

## Current Architecture

### Timeline PCAP Flow (Existing)

```
Client → GET /timelines/calls/{call_id}/pcap
  → bin-api-manager: auth, resolve call → channel → SIP Call-ID
  → bin-timeline-manager (RabbitMQ RPC):
      → Homer API: fetch SIP pcap (hepid=1)
      → Homer API: fetch RTCP pcap (hepid=5)
      → mergePcaps(sip, rtcp) — sorted by timestamp
      → filterInternalPackets() — remove RFC 1918 internal-to-internal
      → return merged pcap bytes (base64 in JSON)
  → bin-api-manager: decode base64 → binary pcap response
```

### RTP PCAP Storage (Existing, Separate)

```
Customer RTPDebug=true → bin-call-manager sends "start recording" to RTPEngine
  → RTPEngine writes pcap files to local disk
  → voip-rtpengine-proxy pcap watcher detects metadata files
  → Uploads pcap to GCS: gs://<bucket>/rtp-recordings/<sip-call-id>-<ssrc>.pcap
  → Cleans up local files
```

Key facts:
- RTP pcap filenames follow `<sip-call-id>-<ssrc>.pcap` convention
- A typical two-party call produces 2+ pcap files (one per RTP direction/SSRC)
- Re-INVITEs, hold/resume, or codec changes can produce additional files
- Files use `LinkTypeEthernet` (same as Homer pcaps)

## Approach

Extend `bin-timeline-manager` to also fetch RTP pcap files from GCS, merge them with the SIP/RTCP pcaps, and return the combined result. No API changes needed.

### Updated Data Flow

```
GetPcap(callID, sipCallID, from, to)
    ├── Homer API: fetch SIP pcap (hepid=1)        ─┐
    ├── Homer API: fetch RTCP pcap (hepid=5)         ├── parallel
    └── GCS: list + download rtp-recordings/<sipCallID>-*.pcap ─┘
            ↓
    Write GCS downloads to temp files
            ↓
    K-way merge all pcap sources by timestamp (file-based readers)
            ↓
    Filter internal-to-internal IP packets
            ↓
    Return merged pcap bytes
```

## Design Decisions

### 1. GCS client lives in timeline-manager

Timeline-manager fetches RTP pcaps directly from GCS. This avoids adding new RPC hops through other services. The trade-off is adding a GCS client dependency to timeline-manager, but this is a standard infrastructure dependency already used by voip-rtpengine-proxy.

### 2. Always attempt GCS lookup (unconditional)

No opt-in flag or query parameter. The endpoint always checks GCS for RTP pcaps. If none found (most calls won't have them since RTPDebug is opt-in), it returns the SIP-only pcap as today. The GCS prefix listing is cheap and returns immediately when no files match.

### 3. File-based merge (not in-memory)

Downloaded RTP pcap files are written to temp files on disk. The merge reads one packet at a time from each source using `pcapgo.Reader` on file handles, picks the earliest timestamp (k-way merge sort), and writes to the output buffer. This keeps memory usage constant regardless of total pcap size, and removes the need for an arbitrary download size limit.

Temp files are cleaned up with `defer os.Remove()` after the response.

### 4. Multi-source merge function

Add a new `mergeMultiplePcaps(sources []io.Reader) ([]byte, error)` that handles N pcap sources. The existing `mergePcaps` can delegate to it by wrapping its `[]byte` inputs in `bytes.NewReader`.

The function:
- Opens a `pcapgo.Reader` on each source
- Validates all sources share the same link-layer type (excludes mismatched sources with a warning log)
- Uses the **maximum snaplen** across all sources for the output header (prevents silent packet truncation)
- Uses a simple linear scan to pick the packet with the earliest timestamp (sufficient for 4-8 sources)
- Writes packets in order to a single output

### 4a. GCSReader interface for testability

Define a small interface in `pkg/siphandler/gcsreader.go` so GCS operations are mockable in tests:

```go
type GCSReader interface {
    ListObjects(ctx context.Context, prefix string) ([]string, error)
    DownloadObject(ctx context.Context, objectPath string, dest io.Writer) error
}
```

The real implementation wraps `*storage.Client`. Tests use a mock generated by `go generate`.

### 5. Same internal IP filter applied to RTP packets

RTP packets are filtered with the same RFC 1918 rule as SIP packets. Packets where both source and destination are private IPs are removed to avoid exposing internal infrastructure topology.

### 6. Best-effort GCS integration (graceful degradation)

- GCS unreachable → log warning, return SIP+RTCP only
- Individual file download failure → skip that file, merge what we have
- Link-layer type mismatch between Homer and RTP pcaps → log warning, return SIP+RTCP only
- All GCS files empty → same as no RTP pcaps found

No degradation from current behavior — the endpoint never returns less data than it does today.

### 7. Parallel downloads with timeout

All GCS file downloads run concurrently (goroutines). A 30-second **sub-context** (derived from the parent via `context.WithTimeout`) bounds the total GCS operation time (list + all downloads). This sub-context is independent from the parent context used for Homer API calls, so a GCS timeout does not cancel Homer fetches.

If the timeout fires, whatever files completed are used; the rest are skipped.

### 8. Debug logging per project conventions

Add `log.WithField(...).Debugf(...)` after every successful GCS list and download operation, following the monorepo's mandatory debug logging pattern for retrieved data.

## GCS Object Lookup

To find RTP pcap files for a call:

1. Build prefix: `rtp-recordings/<sip-call-id>-`
2. List objects in bucket with that prefix
3. Download all matching `.pcap` files in parallel
4. Each file is written to a temp file for file-based merge

The trailing `-` after the SIP Call-ID prevents false matches against longer Call-IDs.

## Merge Algorithm

```
func mergeMultiplePcaps(sources []io.Reader) ([]byte, error):
    1. Open pcapgo.Reader on each source
    2. Read first packet from each reader into a buffer (one packet per source)
    3. Determine shared link-layer type from first valid source
       - Exclude sources with mismatched link-layer types (log warning)
    4. Use max(snaplen) across all included sources
    5. Create pcapgo.Writer with the shared link-layer type and max snaplen
    6. Loop:
       a. Find the buffered packet with the earliest timestamp (linear scan)
       b. Write it to output
       c. Read next packet from that source (or mark source as exhausted)
       d. Repeat until all sources exhausted
    7. Return output bytes
```

For the number of sources we expect (typically 4-8: SIP + RTCP + 2-6 RTP files), a simple linear scan to find the minimum timestamp is sufficient. A min-heap would be premature optimization.

## Error Handling

| Scenario | Behavior |
|----------|----------|
| `GCS_BUCKET_NAME` not set | GCS integration disabled, return SIP+RTCP only |
| GCS list fails | Log warning, return SIP+RTCP only |
| Individual download fails | Skip file, merge remaining sources |
| Download timeout (30s) | Use completed downloads, skip timed-out ones |
| Link-layer type mismatch | Log warning, exclude mismatched sources (merge remaining) |
| No RTP pcaps in GCS | Return SIP+RTCP only (no change from today) |
| Temp file creation fails | Log error, skip GCS data, return SIP+RTCP only |

## Scope Exclusions

- **Multiple call legs**: The endpoint currently resolves a single channel/SIP Call-ID per call. Multi-leg support is out of scope for this feature.
- **New API endpoints**: No new endpoints or query parameters. Same endpoint, richer content.
- **RTP pcap retention/lifecycle**: No changes to how voip-rtpengine-proxy uploads or manages pcap files.
- **OpenAPI spec changes**: Response format unchanged (binary pcap), no spec update needed.

## Testing Strategy

- **Unit tests for `mergeMultiplePcaps`**: 0 sources (empty), 1 source (passthrough), 2 sources (backward compat with existing `mergePcaps` behavior), 3+ sources, mixed empty sources
- **Mock GCSReader tests**: Verify GCS list + download flow, mock returns 0/1/N files
- **Link-layer mismatch test**: Verify mismatched sources are excluded (not a hard failure), remaining sources still merge
- **Timeout test**: Verify partial results when some downloads time out
- **Empty GCS test**: Verify no regression when no RTP files exist (GCS returns empty list)
- **GCS disabled test**: Verify `GCS_BUCKET_NAME=""` produces identical behavior to current code
- **Filter test**: Verify internal IP filtering applies to RTP packets
- **Snaplen test**: Verify max snaplen is used when sources have different values

## Operational Considerations

- **K8s memory limit**: Current limit is 30Mi (`k8s/deployment.yml`). File-based merge keeps memory bounded (one packet per source in flight), but the final output buffer and GCS download I/O still consume memory. Monitor after deployment; may need to increase limit for calls with many RTP streams.
- **Temp file cleanup**: Use `os.CreateTemp("", "rtp-pcap-*.pcap")` with `defer os.Remove()`. On OOM kill, temp files may be orphaned; container restart cleans `/tmp` in ephemeral storage.
- **No caching**: Each request re-downloads from GCS. Acceptable for a diagnostic tool; note as a follow-up optimization if usage is high.
- **GCS auth**: Uses Application Default Credentials (ADC), same as `voip-rtpengine-proxy`. Requires Workload Identity in GKE to grant the timeline-manager service account `storage.objects.get` and `storage.objects.list` on the bucket.
- **End-to-end latency budget**: Homer HTTP timeout is 30s. GCS sub-context timeout is 30s. In worst case (both timeout), total wall-clock time could reach ~60s. The RabbitMQ RPC caller timeout should be reviewed to ensure it accommodates this.
